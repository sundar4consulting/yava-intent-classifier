Intent Configuration Documentation

ℹ️ Note: This document provides a comprehensive guide to managing intents in the YAVA Intent Classifier system. The configuration is stored in the /config directory and supports dynamic updates via Excel uploads and API endpoints.

Configuration Overview

The intent classifier uses a JSON-based configuration system that can be updated without redeploying the agent. Key components:

  • intents_config.json: Main configuration file containing all intent definitions
  • config_loader.py: Handles loading and reloading configuration
  • excel_parser.py: Parses Excel files into JSON format
  • admin_api.py: Provides REST endpoints for configuration management

Excel Template: intent_business_capture_requirement_template.html

The Excel template (intent_business_capture_requirement_template.html) is a structured form for capturing business requirements for new intents. It ensures all necessary information is collected before adding intents to the system.

Template Structure

The template includes the following sections:

  1. Intent Metadata
    • Intent ID (unique identifier)
    • Intent Name (human-readable name)
    • Category (healthcare, benefits, claims, etc.)
    • Agent Routing (target agent name)
    • Priority (1-5, higher = more important)
  2. Business Requirements
    • Description (short summary)
    • Use Cases (when this intent should trigger)
    • Expected Outcomes (what the agent should do)
  3. Training Data
    • Training Utterances (sample user messages)
    • Keywords (important terms)
    • Disambiguation Prompt (clarification questions)
  4. Validation Rules
    • Confidence Thresholds
    • Fallback Behavior

Steps to Add Intents

Method 1: Excel Upload (Recommended for Bulk Updates)

  1. Prepare Excel File
    • Download and fill out intent_business_capture_requirement_template.html
    • Ensure all required columns are populated
    • Save as .xlsx format
  2. Upload via Admin API
    curl -X POST "http://localhost:8000/admin/upload-excel" \ -H "Content-Type: multipart/form-data" \ -F "file=@your_intent_config.xlsx"
  3. Reload Configuration
    curl -X POST "http://localhost:8000/admin/reload-config"
  4. Validate Configuration
    curl "http://localhost:8000/admin/validate-config"

Method 2: Direct API Addition (For Single Intents)

curl -X POST "http://localhost:8000/admin/add-intent" \ -H "Content-Type: application/json" \ -d '{ "intent_id": "INT-PHR-0001", "intent_name": "pharmacy", "category": "healthcare", "agent_routing": "PharmacyAgent", "priority": 4, "description_short": "Handle pharmacy-related inquiries", "disambiguation_prompt": "Are you asking about prescription refills, pharmacy locations, or medication costs?", "training_utterances": [ "I need to refill my prescription", "Where is the nearest pharmacy?", "How much does my medication cost?" ], "keywords": ["prescription", "pharmacy", "medication", "refill"] }'

Adding Disambiguation

Disambiguation helps when the classifier has low confidence in an intent. It provides clarification questions to better understand user intent.

Steps to Add Disambiguation

  1. Identify Ambiguous Scenarios
    • Review similar intents that might overlap
    • Test with sample utterances that could match multiple intents
  2. Create Disambiguation Prompt
    • Write clear, specific questions
    • Include options when possible
    • Keep it conversational
  3. Add to Configuration
    • Include in Excel template under "Disambiguation Prompt" column
    • Or add via API as shown above

Example Disambiguation Prompts

Detailed Attribute Explanations

This section provides detailed explanations of each attribute, including how they affect system behavior, examples, and best practices.

Priority (1-5)

Description: Indicates the importance level of the intent, where 5 is the highest priority and 1 is the lowest.

Behavior Changes:

  • Routing Priority: Higher priority intents are routed first when multiple intents match with similar confidence scores.
  • Confidence Threshold: Priority 4-5 intents use a lower confidence threshold (0.7) for automatic routing, while priority 1-3 require higher confidence (0.85) to avoid false positives.
  • Disambiguation Trigger: Lower priority intents trigger disambiguation more easily when confidence is borderline.
  • Agent Escalation: Priority 5 intents may escalate to human agents if classification confidence drops below 0.6, while priority 1 intents allow lower confidence acceptance.

Examples:

  • Priority 5: "emergency_medical" - Immediate routing required, low tolerance for misclassification
  • Priority 4: "pharmacy_refill" - Important but not urgent, standard confidence thresholds
  • Priority 3: "general_inquiry" - Routine questions, higher tolerance for clarification
  • Priority 2: "wellness_tips" - Informational, can accept lower confidence
  • Priority 1: "casual_chat" - Fallback intent, very permissive matching

Category

Description: Groups related intents for organizational and analytical purposes.

Behavior Changes:

  • Analytics Grouping: Categories enable reporting and performance analysis by intent type.
  • Agent Specialization: Some agents may specialize in specific categories (e.g., "claims" vs "benefits").
  • Training Optimization: Category-based training helps the model learn domain-specific patterns.
  • Business Rules: Categories can trigger different business logic or compliance requirements.

Examples:

  • healthcare: Medical appointments, prescriptions, health records
  • claims: Filing claims, claim status, appeals
  • benefits: Coverage details, plan information, enrollment
  • billing: Payments, statements, premium questions
  • wellness: Fitness programs, health tips, preventive care

Agent Routing

Description: Specifies which specialized agent should handle the intent.

Behavior Changes:

  • Agent Selection: Determines which agent receives the conversation after classification.
  • Context Transfer: Relevant conversation history and user data are passed to the routed agent.
  • Skill Activation: Different agents have different tools and capabilities based on their specialization.
  • Fallback Handling: If the specified agent is unavailable, the system may route to a general agent.

Examples:

  • PharmacyAgent: Handles prescription refills, medication questions, pharmacy locations
  • ClaimsAgent: Processes claim filings, status checks, appeal assistance
  • BenefitsAgent: Explains coverage, answers plan questions, handles enrollment
  • WellnessAgent: Provides fitness advice, wellness program information

Description Short

Description: A brief, human-readable summary of what the intent handles.

Behavior Changes:

  • Documentation: Used in admin interfaces and reporting for intent identification.
  • Debugging: Helps developers understand intent purpose during troubleshooting.
  • Analytics: Provides context for performance metrics and user interaction analysis.
  • Training: Guides the creation of appropriate training utterances and keywords.

Best Practices:

  • Keep under 100 characters
  • Use action-oriented language (e.g., "Handle pharmacy inquiries" vs "Pharmacy stuff")
  • Include key outcomes (e.g., "Process prescription refills" vs "Prescription questions")

Disambiguation Prompt

Description: A question or set of options presented to users when the system has low confidence in intent classification.

Behavior Changes:

  • Confidence Threshold: Triggers when classification confidence falls below 0.75 for priority intents or 0.8 for lower priority ones.
  • User Experience: Provides clear options instead of guessing, improving accuracy and user satisfaction.
  • Conversation Flow: Pauses automated routing and waits for user clarification before proceeding.
  • Analytics Tracking: Disambiguation events are logged for model improvement and intent refinement.

Best Practices:

  • Present 2-4 clear options when possible
  • Use conversational language
  • Avoid technical jargon
  • Test prompts with actual users for clarity

Examples:

  • Good: "Are you looking to: 1) Refill a prescription, 2) Find a pharmacy location, or 3) Check medication costs?"
  • Poor: "Please clarify your pharmacy-related question."

Training Utterances

Description: Sample user messages that should trigger this intent, used to train the classification model.

Behavior Changes:

  • Model Training: Directly influences how the AI learns to recognize intent patterns.
  • Confidence Scoring: More diverse utterances improve classification accuracy and confidence scores.
  • Edge Case Handling: Varied examples help the model handle different phrasings and contexts.
  • Performance Metrics: Quality and quantity of utterances affect precision, recall, and F1 scores.

Best Practices:

  • Provide 10+ diverse examples per intent
  • Include variations in length, formality, and wording
  • Cover both direct and indirect ways users might express the intent
  • Include common misspellings or abbreviations
  • Avoid duplicate or very similar phrases

Examples for "pharmacy" intent:

  • "I need to refill my prescription"
  • "Where can I pick up my meds?"
  • "How much does my prescription cost?"
  • "Find the closest pharmacy"
  • "I ran out of my medication"

Keywords

Description: Important terms that strongly indicate this intent, used to boost matching confidence.

Behavior Changes:

  • Matching Boost: Presence of keywords increases classification confidence by 10-20%.
  • Feature Engineering: Keywords are used as features in the classification algorithm.
  • Search Optimization: Keywords help in semantic search and vector similarity calculations.
  • Intent Discrimination: Helps distinguish between similar intents (e.g., "prescription" vs "appointment").

Best Practices:

  • Include 3-8 key terms per intent
  • Add synonyms and variations (e.g., "prescription", "rx", "medication")
  • Focus on domain-specific terminology
  • Include common misspellings if relevant
  • Update based on user interaction analysis

Examples:

  • Pharmacy: prescription, refill, medication, pharmacy, rx, meds
  • Claims: claim, reimbursement, payment, coverage, filed, status
  • Benefits: coverage, plan, benefits, deductible, copay, premium

Intent ID

Description: A unique identifier for each intent following a standardized naming convention.

Behavior Changes:

  • System Identification: Used internally by the system to reference and manage intents uniquely.
  • API Endpoints: Serves as the primary key for intent-related API operations and database lookups.
  • Version Control: Enables tracking of intent changes and maintaining audit trails.
  • Integration: Facilitates integration with external systems that reference intents by ID.

Best Practices:

  • Use format: INT-CATEGORY-NNNN (e.g., INT-PHR-0001, INT-CLM-0047)
  • CATEGORY should be a 3-letter abbreviation (PHR=Pharmacy, CLM=Claims, BEN=Benefits, etc.)
  • Increment numbers sequentially within each category
  • Never reuse IDs, even for deleted intents

Examples:

  • INT-PHR-0001: pharmacy - First pharmacy-related intent
  • INT-CLM-0047: claims_status - 47th claims-related intent
  • INT-BEN-0023: benefits_enrollment - 23rd benefits-related intent

Intent Name

Description: A human-readable, machine-friendly name for the intent used in code and APIs.

Behavior Changes:

  • API Responses: Returned in classification results and used by downstream systems.
  • Agent Logic: Used by agents to determine which tools and responses to activate.
  • Analytics: Appears in reports and dashboards for intent performance tracking.
  • Configuration: Referenced in agent routing rules and business logic.

Best Practices:

  • Use camelCase or snake_case (e.g., pharmacyRefill, claims_status)
  • Keep names descriptive but concise (under 30 characters)
  • Avoid spaces and special characters except underscores
  • Make names unique across all intents
  • Use consistent naming patterns within categories

Examples:

  • pharmacy: General pharmacy inquiries
  • prescription_refill: Specific prescription refill requests
  • claims_status: Checking claim processing status
  • benefits_summary: Requesting benefits overview

Use Cases

Description: Describes the specific scenarios and situations when this intent should be triggered.

Behavior Changes:

  • Training Guidance: Helps create more targeted training utterances and keywords.
  • Validation: Used during intent review to ensure coverage of intended scenarios.
  • Documentation: Provides context for business users and developers about intent scope.
  • Testing: Guides the creation of test cases to validate intent performance.

Best Practices:

  • Be specific about user scenarios and contexts
  • Include both common and edge cases
  • Use action-oriented descriptions
  • Consider user intent rather than just keywords
  • Keep under 200 characters for readability

Examples:

  • Pharmacy: "When users need prescription refills, pharmacy locations, or medication information"
  • Claims: "When users want to file new claims, check claim status, or appeal denied claims"
  • Benefits: "When users inquire about coverage details, plan information, or enrollment processes"
  • Emergency: "When users report urgent medical situations requiring immediate assistance"

Expected Outcomes

Description: Defines what the system should accomplish when this intent is correctly identified.

Behavior Changes:

  • Agent Behavior: Guides the specific actions and responses the routed agent should provide.
  • Success Metrics: Defines what constitutes successful intent handling for analytics.
  • Quality Assurance: Used to evaluate whether the intent was handled appropriately.
  • Process Flow: Helps design the conversation flow and escalation paths.

Best Practices:

  • Focus on user outcomes, not just system actions
  • Be specific about deliverables or resolutions
  • Include escalation paths for complex cases
  • Consider both automated and human-assisted outcomes
  • Align with business objectives and user needs

Examples:

  • Pharmacy: "User receives prescription refill confirmation and pickup instructions"
  • Claims: "User gets claim status update and next steps for processing"
  • Benefits: "User receives personalized benefits summary and coverage details"
  • Emergency: "User is immediately connected to emergency services or medical professionals"

Troubleshooting

  • Upload Fails: Check Excel format and required columns
  • Validation Errors: Review error messages and fix missing fields
  • Classification Issues: Add more training utterances or adjust disambiguation
  • Reload Problems: Check file permissions and JSON syntax
